# Billing v1.1 — Program Roadmap

**Date:** 2026-06-21
**Status:** Roadmap (decomposition + sequencing). Each sub-project gets its own spec → plan → implementation cycle.
**Predecessor:** The 4-part Billing v1.0 program (Catalog #1365, Invoice Engine #1383, Recurring Contracts #1411, Stripe Payments #1422 — all merged). Frame: `docs/superpowers/specs/billing/2026-06-14-billing-architecture-overview.md`.

## Why this exists

Billing v1.0 shipped the engine but left a recorded backlog of deferred cleanup (enum single-source-of-truth, a net-new test backlog) plus several editing/workflow gaps that surfaced in use: invoices and contracts have no in-place line editing (remove-then-re-add only), no bulk list operations, no approval gate, and no first-class way to correct an *issued* document. There's also a planned-but-unverified "Contact Info & Terms" feature (2026-06-19) and a handful of UX papercuts (#1418).

This roadmap collapses all of that into one sequenced program — **Billing v1.1** — so the work is ordered by risk and dependency rather than picked off ad hoc. It is explicitly **not** a single spec: it's ~6–8 PRs across schema, API, UI, and accounting-sensitive immutability rules.

## Guiding constraints

- **Issued financial documents are immutable.** Corrections to an issued invoice happen via credit note / void+reissue, never by mutating the issued row. This decision (made up front) governs E4 and bounds E1/E3. The engine already leans this way: `assertDraft` gating, deterministic `deriveInvoiceStatus`, one-directional guards (can't un-void, can't un-issue) — see `invoiceService.ts:39-41,521-614` and `invoiceMath.ts:60-76`.
- **Foundation before features.** The enum refactor and the regression test net land *before* the editing/workflow work that depends on them — unifying the status/method/source enums while five callers are actively churning them is worse than doing it first.
- **Each sub-project follows the repo's normal discipline:** RLS shape + cascade contract for any new table, TDD, integration tests in `src/__tests__/integration/*.integration.test.ts`, idempotent dated migrations.

## Sub-projects

### Foundation

#### F1 · Enum single-source-of-truth + UX cleanup
*Size: S. Risk: low. Sequence: first.*

Collapse the duplicated billing enums into one canonical tuple set in `packages/shared`, consumed everywhere:
- `InvoiceStatus` — `['draft','sent','partially_paid','overdue','paid','void']` — defined at both `apps/api/src/db/schema/invoices.ts:9-11` (pgEnum) and `apps/api/src/services/invoiceTypes.ts:1-2` (TS union).
- `PaymentMethod` — `['cash','check','bank_transfer','card','other']` — `invoices.ts:15-17` + `invoiceTypes.ts:5-6`.
- `InvoiceLineSourceType` — `['time_entry','part','catalog','bundle','manual','contract']` — `invoices.ts:12-14` + `invoiceTypes.ts:3-4`.
- Zod enums in `packages/shared/src/validators/invoices.ts:57,70` (status + method) duplicate the same values.
- Plus the web and portal copies (`apps/web`, `apps/portal`) that hand-repeat these literals.

Approach: define `const`-asserted tuples in `packages/shared` as the source; derive the Drizzle `pgEnum`, the TS unions, and the Zod `z.enum` from them. **No DB migration** — pgEnum values are unchanged; this is a code-consolidation refactor, behavior-neutral. Bundle the cheap UX papercut(s): the stale "Draft" header after issue (#1418) and any residual routing/label drift.

*Why first:* E3 (workflow) and E4 (credit notes) both edit these enums; unify before they churn.

> **Note:** quotes, contracts, and catalog carry their own separate enums (`quotes.ts:11-16`, `contracts.ts:8-16`, `catalog.ts:9-11`). They are **out of scope** for F1 — F1 only unifies the three invoice/payment enums that are genuinely *duplicated* within the invoice domain. Don't gold-plate by hauling the sibling domains in.

#### F2 · Test backlog (regression net)
*Size: M. Risk: low. Sequence: second (overlaps F1; some cases pair with E3/E4).*

Net-new coverage on *existing shipped* behavior, so the riskier editing work below has a safety net. Recorded + survey-confirmed gaps:
- **Numbering under concurrency** — gapless partner-scoped allocation (`invoiceNumbers.ts`) under parallel issue.
- **Cross-partner list isolation at the service layer** — not just route/RLS; the service-level filter.
- **`voidPayment` reversal** — `invoiceService.ts:569-589` (manual) and the shared shape with `reflectStripeRefund` (`stripeReconcile.ts:109-185`); currently only the Stripe path is well-covered (`stripeReconcile.test.ts`).
- **Tax-exempt / non-zero / partner-default-rate through `issueInvoice`.**
- **Illegal-transition branches** — the guards at `invoiceService.ts:538,613,614`; recompute-after-payment integration flow (`recomputeInvoiceStatus`, currently only unit-covered in `invoiceMath.test.ts`).
- **Validator negatives** — `taxRate>1`, negative amounts, etc.
- **`aiToolsBilling`** — `aiToolsBilling.ts:106` (`list_invoices`/`get_invoice`) has **no test**; cover registration, input validation, org-scope guard, error-to-JSON conversion.
- **`markViewed` write-once**, web `InvoiceWorkspace` / portal components, assembly SQL filters (`invoiceAssembly.ts`).

Real-DB cases go in the integration suite per `test_realdb_placement_convention`.

### Editing & Workflow

#### E1 · Inline draft line editing
*Size: M. Risk: low-med. Sequence: after F1/F2.*

`PATCH /lines/:lineId` for **draft** invoices (and the parallel gap on contracts/quotes), with in-place edit UI (quantity / unit price / description) instead of remove-then-re-add. Drafts are already mutable (`assertDraft`), so this is additive. Must re-resolve totals via `invoiceMath` and respect the same draft-only guard. Watch the **billing UI↔Zod payload trap** (money as strings, optionals as `undefined` not `null`) — assert the *actual* browser payload in tests.

#### E2 · Bulk actions
*Size: M. Risk: med. Sequence: after E1.*

Multi-select on the invoices/quotes list: bulk issue, bulk send, bulk mark-paid, bulk void, bulk export. Each bulk op is a loop over the existing single-item guarded path (no new transition semantics) with **partial-success reporting** — surfaced via `runAction`'s aggregate handling, recorded in `runActionAllowlist.ts` if needed. Per-item RLS still enforced; no list-wide privileged path.

#### E3 · Approval + status workflow + audit timeline
*Size: L. Risk: med-high. Sequence: after E2.*

A review/approval gate before issue/send, hardened/explicit transitions, and a per-document history timeline. This is where the deterministic `deriveInvoiceStatus` may need an explicit `pending_approval`-style state and a real transition guard rather than pure derivation. Touches the F1 enums directly (hence F1 first). Audit timeline reads from existing billing events (`invoiceEvents.ts`) + audit log.

#### E4 · Editing issued/sent documents (credit notes)
*Size: L. Risk: high. Sequence: last.*

**Decision locked: immutable + credit notes.** Issued invoices are never mutated. "Editing" an issued document means issuing a **credit note** (and optionally a corrected reissue), preserving the original as an immutable audit record. New `credit_notes` table (RLS shape-1 + cascade contract + numbering, mirroring invoices), partner-scoped gapless numbering, PDF + portal surfacing, and reconciliation into the invoice balance. Most accounting-sensitive piece; gets its own full brainstorm when reached. **Open question deferred to E4 spec:** whether credit notes reduce an existing invoice's balance in place or stand as independent documents that net at the statement level.

### Feature

#### D1 · Contact Info & Terms — *verify, then finish*
*Size: S–M (mostly verification). Risk: low. Sequence: flexible — can run alongside F2/E1.*

A spec **and** a full implementation plan already exist (`docs/superpowers/specs/billing/2026-06-19-quotes-invoices-contact-info-design.md`, plan `docs/superpowers/plans/billing/2026-06-19-quotes-invoices-contact-info.md`), and the backend is **at least partially built**: `invoiceSettingsRoutes` is mounted (`apps/api/src/index.ts:773-774`), `updatePartnerBillingSettings`/`updateOrgBillingSettings` exist, and `billing-contact-info.integration.test.ts` is present — yet the plan's checkboxes are all unticked.

So D1 is **not** greenfield. Step one is a status audit: diff what actually shipped against the 13-step plan (partner/invoice/quote columns + migration, validators, snapshot-at-issue/send, PDF From-block + T&C rendering, web UI). Then finish whatever's missing and close the plan out — do **not** re-implement what's already merged.

## Recommended sequence

```
F1 (enum SSOT + UX) ──┐
                      ├─► E1 (inline edit) ─► E2 (bulk) ─► E3 (approval/workflow) ─► E4 (credit notes)
F2 (test net) ────────┘
D1 (contact info: verify→finish) ── runs in parallel, low dependency
```

Rationale: F1 + F2 are the foundation (unify enums, lay the regression net) and can overlap. The editing/workflow chain is strictly ordered by risk — drafts (E1) before lists (E2) before transition semantics (E3) before immutable-doc corrections (E4). D1 is independent and can slot wherever capacity allows.

## What's explicitly out of scope (YAGNI)

- Unifying the quote/contract/catalog enums (separate domains, not duplicated within themselves).
- Proration / overage / fixed-term renewal on contracts (already deferred to contracts v2).
- Mutating issued invoices in place / un-issue / re-open (rejected in favor of credit notes).
- Write-capable billing AI tools (read-only stays for now).
